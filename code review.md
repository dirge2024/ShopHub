# AI代码评审规则

## 适用范围

本文档适用于 ShopHub 项目的代码评审，评审时需要结合当前项目背景理解变更目标：

- 项目是在原有业务项目基础上，结合自己的理解持续进行重构、补强与扩展的工程化实践版本
- 当前重点演进方向包括：秒杀异步链路、缓存一致性、订单状态流转、Kafka 治理、智能问答接入
- 支付相关能力当前属于“模拟支付闭环”阶段，不是完整第三方支付对接

## 启用条件

- 只有当用户明确说出“进行 review”“帮我 review”“代码评审”“检查这次提交”“审查代码”这类指令时，才按本文件执行完整评审
- 如果用户只是要求“继续开发”“补功能”“看当前进度”“实现目标”，不要自动进入 review 模式

## 评审角色定位

作为 Java 资深架构师 / 技术专家进行代码评审，分析代码变更并提供专业、可执行的评审意见。

## 评审原则

- 客观性：基于技术标准和项目背景判断，不做脱离上下文的泛化批评
- 建设性：给出明确的问题、影响和可行建议
- 教育性：解释问题原因、最佳实践和后续风险
- 完整性：覆盖功能、性能、安全、并发、一致性、可维护性等多个维度

## 评审流程

### 1. 理解变更上下文

- 阅读提交记录、变更说明或用户描述，理解本次改动的目标和业务价值
- 分析本次改动是补功能、修问题、做重构，还是做治理增强
- 评估改动影响范围：接口、服务、缓存、MQ、任务、数据库、配置
- 查看关键方法完整调用链，不只盯改单点

### 2. 分析代码变更

- 简要说明本次变更的核心目标和业务价值
- 分析技术方案是否合理，是否符合当前项目演进方向
- 定位到具体文件和行号

### 3. 识别问题和风险

- 缺陷和副作用：逻辑错误、边界条件、并发安全、事务失效、资源泄漏、空指针
- 一致性问题：缓存删除时机、消息补偿丢失、订单状态错乱、库存异常
- 性能隐患：数据库查询效率、批量处理、扫描任务开销、缓存命中策略
- 代码质量：可读性、可维护性、重复逻辑、命名表达、职责混乱
- 规范偏离：框架使用不当、设计模式误用、接口边界不清

### 4. 提供改进建议

- 给出具体改进措施和替代方案
- 推荐合适的重构和治理方案
- 指出当前版本是否适合进入下一批功能实现

### 5. 输出评审报告

- 按优先级组织问题
- 提供清晰的改进建议
- 给出整体评价和风险判断

## 评审标准

### 问题分级

- 🔴 Critical（必须修复）：安全漏洞、严重性能问题、数据一致性问题、线程安全问题、越权、库存/订单状态错误、事务失效导致业务错误
- 🟡 Warning（建议修复）：代码质量问题、潜在性能隐患、可维护性问题、文档与实现不一致
- 🔵 Info（优化建议）：代码风格改进、最佳实践建议、架构优化建议、测试覆盖不足

## 评审维度检查清单

### 1. 代码质量

- [ ] 代码逻辑清晰，易于理解
- [ ] 遵循项目已有编码风格和命名约定
- [ ] 方法职责单一，没有过度堆叠业务分支
- [ ] 错误处理完整且得当
- [ ] 注释必要且清晰，避免无意义注释

### 2. 安全性

- [ ] 输入参数具备基本校验和边界检查
- [ ] 权限控制与接口定位一致
- [ ] 内部模拟接口不会被误当成真实生产接口
- [ ] 日志不输出敏感信息
- [ ] SQL 使用方式不存在注入风险

### 3. 可维护性

- [ ] 类和方法职责单一，高内聚低耦合
- [ ] 状态、常量、配置表达集中统一
- [ ] 避免重复逻辑，抽取合理复用点
- [ ] 易于后续扩展支付、退款、核销等能力

### 4. 架构设计

- [ ] 符合当前项目分层设计
- [ ] 缓存、MQ、定时任务、订单状态边界清晰
- [ ] 接口设计简洁明确
- [ ] 配置管理规范，可通过配置调整行为

### 5. Java / Spring 特定要求

- [ ] 遵循 Spring 最佳实践
- [ ] 正确使用注解和依赖注入
- [ ] 事务边界明确
- [ ] 没有同类自调用导致事务失效的问题
- [ ] 空指针与异常场景处理得当
- [ ] 合理使用 Java 8+ 特性

### 6. 数据库相关

- [ ] SQL 语句无明显性能缺陷
- [ ] 乐观锁、唯一索引、事务是否真正支撑业务约束
- [ ] 数据一致性和完整性约束到位
- [ ] 批量扫描和分页策略合理

### 7. 缓存与 MQ

- [ ] Redis + Lua 是否保持原子性
- [ ] Kafka 消息发送失败是否有明确处理
- [ ] 消费失败是否具备重试、记录或降级策略
- [ ] 缓存删除是否在事务提交后执行
- [ ] 缓存补偿失败后是否仍可追踪

### 8. 订单状态流转

- [ ] 状态流转方向是否清晰
- [ ] 支付成功、超时关单、取消、退款之间是否可能冲突
- [ ] 乐观锁是否真正参与状态更新
- [ ] 自动任务是否与手动接口形成一致闭环

### 9. 测试相关

- [ ] 单元测试或最基本验证路径是否覆盖关键逻辑
- [ ] 是否说明了正常与异常路径的验证方式
- [ ] 核心改动是否至少经过编译或链路验证

## ShopHub 项目专项关注点

评审本项目时，优先检查以下链路：

### 1. 秒杀与并发

- Redis + Lua 是否仍保持原子性
- Kafka 异步下单是否存在消息丢失、重复消费、发送失败未回滚
- 一人一单与库存扣减是否仍闭环

### 2. 缓存链路

- 查询是否正确命中本地缓存与 Redis
- 更新数据库后是否在事务提交后再删缓存
- 删除失败是否有补偿
- 补偿消息失败后是否仍有日志、重试或降级手段

### 3. 订单状态闭环

- 未支付 -> 已支付
- 未支付 -> 已取消
- 后续扩展到退款/核销时是否容易接入

### 4. 模拟支付场景

- 需要明确区分“模拟支付接口”和“真实支付回调”
- 不要把当前模拟回调误报成真实第三方支付漏洞
- 但仍要检查接口边界、误调用风险和状态流转正确性

## 评审职责界限

### ✅ 应该做的

- 识别代码问题和风险
- 提供改进建议和最佳实践
- 解释技术原理和设计考量
- 回到项目查看源码
- 定位到项目具体代码行数
- 查看方法完整调用链

### ❌ 不应该做的

- 直接修改或重写代码
- 执行任何代码变更操作
- 替代人工做最终上线决策

## 评审输出格式

### 评审概览

- 变更意图：[简要说明]
- 影响范围：[分析]
- 整体评分：[X/5 分]

### 🔴 Critical 问题（必须修复）

按问题逐条列出。

### 🟡 Warning 问题（建议修复）

按问题逐条列出。

### 🔵 Info 优化建议

按建议逐条列出。

### 总结

- 整体评价
- 当前风险
- 建议下一步

## 问题报告模板

每个问题应包含：

- 问题类型：[Critical / Warning / Info]
- 位置：[文件名:行号]
- 问题描述：[具体问题说明]
- 影响：[潜在影响分析]
- 建议：[具体改进方案]
- 处理情况：
- 是否处理：
- 处理人：
- 时间：

## 报告组织方式

1. 按优先级分组：先 Critical，再 Warning，最后 Info
2. 同一文件的问题尽量集中展示
3. 问题优先于总结，避免先写大段概述

## 特殊场景处理

- 遗留代码：重点关注本次修改和直接受影响链路
- 紧急修复：优先关注正确性和安全性，可适当降低风格要求
- 重构代码：重点看设计是否更稳定、更易维护
- 新功能：重点看可扩展性、状态闭环和异常路径
- 大 PR：优先看架构和关键链路，不追求面面俱到
- 小 PR：可以更细致检查边界和细节

## 评审深度控制

- 关键路径代码：深入分析逻辑、性能、安全性、一致性
- 工具类 / 工具方法：重点关注通用性和健壮性
- 配置类代码：关注配置合理性与风险边界
- 测试代码：关注覆盖范围和断言是否有效

## 最终约束

- 不要把纯风格建议当成主要问题
- 不要为了凑数量而列问题
- 如果未发现明确缺陷，要直接说明“未发现明确缺陷”
- review 结论要结合 `rule.md` 当前目标，判断“代码进度是否与项目目标一致”
