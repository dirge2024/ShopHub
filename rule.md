# ShopHub 功能落地规则

本文档用于约束项目实现范围、分批推进节奏，以及标记当前已经完成的能力。

## 总目标

围绕以下能力逐步补齐工程实现：

- Redis + Lua 完成高并发库存扣减与一人一单校验
- Kafka 异步解耦秒杀下单链路，完成流量削峰
- 数据更新后删除缓存，失败时通过消息队列补偿重试，结合 TTL 保证一致性
- 逻辑过期防缓存击穿，空值缓存防缓存穿透
- Caffeine + Redis 二级缓存提升热点查询性能
- 支付回调与超时关单的并发状态控制，使用乐观锁保证准确性
- 接入阿里云百炼大模型，支持 Redis 会话记忆与 Function Calling 查询/预约

## 开发节奏说明

本项目采用小步快跑、分阶段验证的推进方式，提交记录不要求每次都代表“一个完整功能已经最终完成”。

- 允许先提交某个能力点的阶段性实现，用于保存进度、便于回滚、方便前后端联调
- 功能代码实现后，不应立即默认标记为“最终完成”，需要结合前后端联调、接口验证、异常场景检查后再确认
- 一个功能从开发到结束，通常会经历“已实现 -> 待联调 -> 联调通过 -> 已完成”几个阶段
- 如果联调过程中发现问题，允许继续基于当前能力点补丁式迭代，而不是强行等到全部稳定后再提交

## 状态标记说明

- `已实现`：后端代码已落地，具备基础能力，但还未经过完整联调验证
- `待联调`：已经进入前后端联调或接口联测阶段，等待确认行为是否符合预期
- `联调通过`：主要接口流程已经联调通过，功能路径基本成立
- `已完成`：代码、联调、基础验证都已通过，可以作为当前阶段性成果对外描述

## 实施约束

- 每次提交后都应同步更新本文件，但状态不要求只能写“已完成”，可以按“已实现 / 待联调 / 联调通过 / 已完成”标记真实进度
- 不允许为了赶进度把“已实现”直接写成“已完成”，只有联调和基础验证通过后才能标记为“已完成”
- 允许围绕同一个能力点进行多次小提交，先实现、再联调、再修正，这是正常开发节奏
- 新增实现需优先与当前项目阶段目标对齐，避免引入无法解释的复杂设计
- 涉及缓存、消息队列、订单状态的修改，必须同时考虑幂等性与异常补偿
- 提交 Git 时使用简短中文提交说明，准确概括本次阶段性改动，不要求每次都声明“某功能最终完成”
- 所有新增或修改的代码注释统一使用中文，避免中英文注释风格混用
